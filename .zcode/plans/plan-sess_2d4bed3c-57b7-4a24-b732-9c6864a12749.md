## Goal

Two fixes:

1. **JWT backward-compat**: Old JWT tokens carry a bare int user id (`"1"`, `"2"`) in the `sub` claim (pre-migration). After migration 036 converted `users.id` from SERIAL→UUID, these tokens fail with `invalid input syntax for type uuid`. Add a `legacy_int_id` column to `users`, preserve the old int during migration, and fall back to it during JWT-based user lookup until all tokens rotate.
2. **Frontend `[object Object]`**: openapi-fetch helpers in `api.ts` do `if (error) throw error` — throwing a plain `{status, message}` object, not an `Error`. Catch blocks check `instanceof Error` (false) and fall back to `String(err)` → `"[object Object]"`. The server error shape is already consistent (`{status:"fail", message:string}`); the frontend just doesn't decode it.

## Fix 1: JWT legacy id fallback

### Migration 036 (alter in place — not yet applied to prod)

The migration must capture the old int id **before** the `ALTER TABLE users ALTER COLUMN id TYPE UUID` step (line 65), because after that the `id` column is UUID and the original int is lost (and `int_to_uuid()` is dropped at the end).

Add after the `DROP DEFAULT` block (line 58) and before the `ALTER ... TYPE UUID` block (line 60):

```sql
-- Preserve the old SERIAL int id so legacy JWT tokens (which carry the int as
-- the "sub" claim) can still resolve the user until all tokens are rotated.
ALTER TABLE users ADD COLUMN legacy_int_id INTEGER;
UPDATE users SET legacy_int_id = id;          -- id is still INTEGER here
ALTER TABLE users ALTER COLUMN legacy_int_id SET NOT NULL;
CREATE UNIQUE INDEX users_legacy_int_id_idx ON users (legacy_int_id);
```

Place this *before* line 65 (`ALTER TABLE users ALTER COLUMN id TYPE UUID USING int_to_uuid(id)`). The `UPDATE ... SET legacy_int_id = id` runs while `id` is still `INTEGER`. New users created post-migration get `legacy_int_id = NULL` (the column must be nullable for new rows, OR we generate a synthetic legacy id). Since the user said "only for jwt fallback until tokens rotate", nullable is correct — new users never have a legacy int and their JWTs already carry UUIDs.

**Correction to the above**: `NOT NULL` would break `CreateUser` (new users have no legacy int). So: `legacy_int_id INTEGER` nullable, no NOT NULL, unique index allows multiple NULLs (Postgres default). New users: `legacy_int_id = NULL`. Old users: `legacy_int_id = <old int>`. This is exactly the fallback scope the user asked for.

### sqlc query + Go code

Add a `GetUserByLegacyIntID` query to `pkg/db/query/users.sql`:
```sql
-- name: GetUserByLegacyIntID :one
SELECT id, allow_editing, google_oauth_user_id, google_oauth_user_name, player_id
FROM users WHERE legacy_int_id = $1;
```
Regenerate sqlc (`sqlc generate` — available via nix).

In `pkg/elo/users.go`, change `GetUserByID` to accept the JWT `sub` and try UUID lookup first, then legacy-int fallback:
- If `GetUser(ctx, id)` succeeds → return (UUID JWT, normal path).
- If it fails with an invalid-UUID error (the `sub` is an int like `"1"`) → parse the int, call `GetUserByLegacyIntID`. If found → return that user.
- This keeps the fallback localized to the single `GetUserByID` method that all auth paths (`MustGetCurrentUser` → `RequireEditor`/`RequirePlayerID`/`GetMe`) already use.

`CreateUser` (`users.sql`) gets a new `legacy_int_id` column in the INSERT (sqlc will add `LegacyIntID` to the params). New users pass `LegacyIntID: nil` (no legacy id). The `db.User` struct gains a `LegacyIntID *int32` field via sqlc — harmless, unused by response code.

### Why not normalize the JWT `sub` at `ValidateToken`?

The `int_to_uuid` mapping (`00000000-0000-0000-0000-` + zero-padded hex) could be reimplemented in Go and applied in `ValidateToken`. But the user explicitly asked for a `legacy_int_id` column + DB fallback, which is more robust (single source of truth in the DB, no duplicated mapping logic, and the column documents the migration history). I'll go with the column approach as requested.

### Removal later

Once all old JWTs have expired/rotated (after the JWT TTL, e.g. 30 days post-deploy), `legacy_int_id`, the index, the fallback query, and the `GetUserByID` fallback branch can be removed in a follow-up. The ADR will note this.

## Fix 2: Frontend error display

### Root-cause fix (centralized, one place)

In `nextjs/app/api.ts`, change every `if (error) throw error` to throw a proper `Error` carrying the server message:
```ts
if (error) throw new Error(error.message ?? `Ошибка ${response.status}`);
```
This requires destructuring `response` from the openapi-fetch result (`const { data, error, response } = await client.POST(...)`). Now every existing `instanceof Error ? err.message : ...` catch block across ~15 components works correctly and shows the real message.

The existing `apiErrorMessage` helper stays for any call site that catches `unknown` without the `instanceof` guard, but the primary fix is making the thrown value an `Error`.

### Why not change the ~15 catch blocks instead?

Changing `throw error` is one surgical change in `api.ts` that fixes all call sites. Changing 15 catch blocks to use `apiErrorMessage` is more invasive and error-prone (easy to miss one). The `throw new Error(...)` approach also keeps the contract clean: API helpers throw `Error`, like every other JS API.

### Backend error shape (already consistent — no change needed)

The investigation confirmed `ErrorResponse` (`responses.go`) and `errorMiddleware` (`main.go`) both emit `{"status":"fail","message":"<string>"}`, matching the `ApiError` OpenAPI schema. The `500 {"message":"ERROR: invalid input syntax..."}` body was already correct — the frontend just wasn't reading `.message`. No backend contract change needed.

I will **not** add server-side validation that maps the Postgres uuid error to a 400 in this change — that's a separate hardening task and the user's reported symptom (`[object Object]`) is fully fixed by the frontend change. The raw Postgres message will now display correctly as text.
